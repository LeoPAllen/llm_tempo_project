LLM TEMPO EXPERIMENT
PAGE-BY-PAGE SCRIPT

Purpose
This file is the working script/specification for the full oTree experiment currently implemented in this project. It is intended to be the easiest document for the research team to edit. The goal is that the team can revise wording, ordering, measures, and treatment structure here, then pass the file back for code updates.

How to use this file
- Treat this file as the human-readable source of truth for participant-facing content and study structure.
- If the research team changes wording, labels, ordering, conditions, or measures here, those changes can then be wired into the code.
- This file includes both:
  participant-facing content
  implementation notes that should not be shown to participants
- Participant-facing labels are intentionally neutral and do not explicitly reveal measurement intent.

High-level flow
1. Consent
2. Practice LLM interaction + attention check
3. Pre-task measures
4. Task rounds in randomized order
5. Post-task measures
6. Demographics
7. Conclusion

PAGE 2. Practice LLM Interaction + Attention Check
App: pre_tasks_measures
Page class: PracticePage

Current content
- Participant-facing kicker: Instructions
- Participant-facing title: Practice Section
- Participants are told they will complete a series of tasks with help from a custom LLM
- They interact with a mock LLM interface rather than seeing static text
- They copy/paste a practice prompt about camouflage
- The mock LLM streams a hallucinated answer stating arctic foxes are grey in winter
- Attention check question:
  Based on the LLM output, what color are arctic foxes in the winter?
- Response options:
  White
  Grey
  Brown
  Green

Design intent
- This page teaches the interaction format
- It also checks whether participants read the displayed LLM output rather than relying on prior knowledge

TASK A. Ultimatum Game
Task id: ultimatum

Task intro / description
- Participant is proposer in an Ultimatum Game
- Total amount is 1000 USD
- Responder accepts or rejects
- If rejected, both get nothing

Pre-LLM response
- Participant-facing kicker: Your Response
- Prompt: how much of the 1000 USD would you offer?
- Allowed range: 0 to 1000

LLM interaction prompt
- Participant-facing kicker: Assistant
- "In an Ultimatum Game, if I am chosen as the proposer, how should I split 1000 USD between myself and my partner?"

LLM output summary
- Notes that low offers are often rejected
- Suggests a 70/30 split

Post-LLM response
- Participant-facing kicker: Your Response
- Same decision question again

Post-task measures
- Participant-facing kicker: Questions
- Participant-facing title: A Few Follow-Up Questions
- Knowledge/comprehension question 1: what happens if responder rejects?
- Knowledge/comprehension question 2: what increases rejection rates?
- AI evaluation items:
  reliance
  confidence
  helpfulness
  usefulness
  trustworthiness
  understanding
  future use likelihood
  use again

Scholarly source notes
- Ultimatum Game paradigm:
  Guth, W., Schmittberger, R., & Schwarze, B. (1982). An experimental analysis of ultimatum bargaining.
- Typical rejection/offer patterns:
  Camerer, C. F. (2003). Behavioral Game Theory: Experiments in Strategic Interaction.
- AI evaluation items:
  These are custom/adapted items. They are not currently tied to a single validated post-task scale in the code.

TASK C. Net Present Value
Task id: npv

Task intro / description
- Value a 10-year government bond
- Face value: 1000 USD
- Coupon rate: 4%
- Semiannual payments
- Discount rate: 3% annually

Pre-LLM response
- Participant-facing kicker: Your Response
- Prompt: estimate the NPV, rounded to nearest whole number
- Numeric free response

LLM interaction prompt
- Participant-facing kicker: Assistant
- "Teach me how to calculate the Net Present Value of a 10-year government bond without solving it for me. The bond has a face value of 1000 USD and a semiannual payment frequency, current discount rate of 3% and a coupon of 4%."

LLM output summary
- Explains discounting each cash flow
- Notes 20 USD coupon payments
- Notes 1.5% per-period discount rate
- Says to sum discounted coupons and face value

Post-LLM response
- Participant-facing kicker: Your Response
- Same NPV estimate again

Post-task measures
- Participant-facing kicker: Questions
- Knowledge/comprehension question 1: coupon payment every six months
- Knowledge/comprehension question 2: annual discount rate used
- Same AI evaluation items as other tasks

Scholarly source notes
- Net Present Value / bond valuation task:
  This is a domain task rather than a direct replication of a single canonical psychology scale.
- Risk literacy / numeracy background for this type of task:
  Cokely, E. T., et al. (2012). Measuring risk literacy: The Berlin Numeracy Test.
- Your draft also referenced:
  Cokely et al. (2018) / Skilled Decision Theory review as conceptual motivation for numeracy and decision quality.

IMPLEMENTATION NOTES

Randomization
- Task order is currently fully randomized across the 6 implemented tasks
- LLM timing treatment is assigned once per participant and held constant across tasks

Treatment specification
- This experiment currently uses a 2x2 timing design:
  Factor 1: start latency
  Factor 2: stream speed
- Treatments are:
  A: Fast start, fast stream
  B: Slow start, fast stream
  C: Fast start, slow stream
  D: Slow start, slow stream

Current LLM timing treatments
- A: Fast start, fast stream
  0.1 second initial delay, 10 tokens per second
- B: Slow start, fast stream
  2.0 second initial delay, 10 tokens per second
- C: Fast start, slow stream
  0.1 second initial delay, 2 tokens per second
- D: Slow start, slow stream
  2.0 second initial delay, 2 tokens per second

Validation rules currently implemented
- Ultimatum response: 0 to 1000
- Dictator response: 0 to 100
- Trolley response: 0 to 100
- Asset allocation totals must sum to exactly 10000

Known content status
- The experiment is now implemented end-to-end
- Some measure batteries are pragmatic placeholders rather than finalized literature replications
- This file should be updated alongside any wording or flow changes in the oTree code
